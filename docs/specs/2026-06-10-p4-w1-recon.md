## Verdict: GO on all three lanes — and two recon-driven design simplifications

**The infrastructure is dramatically more ready than the proposal assumed.** stable-worldmodel is
*vendored* (`third_party/stable-worldmodel`, editable in `.venv`), which makes the env source, the
collection policy, and a full solver suite (CEM/iCEM/MPPI) ours to read and subclass without
upstream risk. The κ knob already exists as a constructor parameter. The W2–4 "crunch block" is
smaller than scheduled.

**Design simplification #1 (uniform demo protocol).** The κ sweep forbids human demos (they exist
only at default physics). Recon shows the clean alternative: **WeakPolicy rollouts for
representation-training data at every κ** (exactly how the LeWM ecosystem generates data) **+
oracle-planner rollouts for successful demonstrations** (lane b) — one generator across all κ, no
human-vs-scripted confound. The 206-episode human set becomes a κ=0 robustness check, not the
primary protocol. (C2's "demo fraction" semantics unchanged — fractions of the successful-demo set.)

**Gaps to build (P4-step1, hours-scale each):** trajectory-collection script (WeakPolicy → tensors);
param-matched plain-CNN ladder configs; rotation-augmentation hook (R-aug); PoR-style probe rig.
**Spec decision left open:** render size — 96 (matches human demos / LeWM data) vs 224 (env
default). Lean 96 for cost; confirm what resolution LeWM's public checkpoint consumed before fixing.

## Lane (b) — κ-knob + demos + λ-gate: GO, knob pre-exists

| Fact | Evidence |
|---|---|
| **`damping` is a constructor param** — `PushT(..., damping=None)`; applied at reset: `self.space.damping = self.damping` | `env.py:32,214,255–256` |
| Default regime hardcoded `space.damping = 0` → velocities fully damped each step = **quasi-static anchor** (pymunk damping = fraction of velocity retained/second; κ→1 = momentum retained = inertial/sliding regime) | `env.py:536` |
| Secondary knobs (need subclassing, damping doesn't): block `mass=1`, `friction=1`, moments | `env.py:371–373,608–702` |
| State set/get for twin trajectories: `reset(options={'state': ...})` + `_set_state`; velocities live in obs `state` | `env.py:280–288,64` |
| **Solver suite vendored**: `swm/solver/{cem,icem,mppi,gd,predictive_sampling}.py` — both the oracle demo generator and the hierarchical planner's short-window optimizer can build on these | `third_party/.../solver/` |

**Design simplification #2 (oracle demos via true-env CEM).** With `reset(options={'state':...})`,
CEM on the *true env* (env as its own model; pymunk is CPU-cheap) is a uniform successful-demo
generator at every κ — no per-κ learned model needed. step12's pose-control planner remains the
fallback.

**λ̂₁(κ) validation-gate plan (P4-step2, runs BEFORE the spine bets on the knob):** κ grid
$\{0, 0.5, 0.8, 0.95, 1.0\}$; per κ, sample base states along WeakPolicy trajectories; twin-trajectory
divergence (state perturbation $\delta_0 \sim 10^{-4}$ via `reset_to_state`, common action sequence,
fit log-divergence slope over pre-contact windows); average finite-time exponents over the state
distribution; moving-block-bootstrap CIs (Prop 10's estimator). **Gate:** $\hat\lambda_1(\kappa)$
monotone + CI-separated across the grid; fallback = two-point contrast (0 vs best-separated κ).
**Contact-noise mitigations (pre-registered):** exclude collision-instant frames; report
per-segment exponents alongside the average; if contact noise drowns the signal, measure on the
block-only autonomous flow (agent parked).

**Caveat to carry into the spec:** `space.damping` is global (agent body too) — PD agent motion
changes with κ. Acceptable (the *task* changes with physics, as intended) but must be stated; PD
gains stay fixed across κ (fairness rule already pre-registered).

**Open item (one targeted query at spec time):** the sweep-flagged "Dynamic Push-T" variant
(2502.12724? 2603.01953?) — confirm whether it dials physics; cite-and-differentiate if so.

## Lane (c) — 3-D co-anchor: lean GO = ManiSkill3 PegInsertionSide-v1 + VN-DGCNN-small

- **ManiSkill3**: native `pointcloud` obs mode; **PegInsertionSide-v1 ships 1,000 successful
  trajectories** (`python -m mani_skill.utils.download_demo "PegInsertionSide-v1"`); SAPIEN engine;
  insertion = rotation-critical (pose precision is the task). Confirm at spec: controller mode with
  rotation dims (`pd_ee_delta_pose`) for H1-original's action-rotation probes; headless Vulkan on
  the GPU box; **macOS prototyping likely unsupported (Linux-first) — the 3-D lane lives on the GPU
  machine from day one.**
- **LIBERO-spatial fallback**: PoR-protocol comparability (they probed LIBERO; 7-DoF OSC actions)
  but RGB-only obs (point clouds via back-projection only) — second choice.
- **Encoder: VN-DGCNN-small** (FlyingGiraffe/vnn + vnn-pc reference code) — exact SO(3)-equivariant
  linear layers, lightweight, fits 1–2 GPU JEPA training on 1k demos. e3nn-lite = heavier second
  option; EquiformerV2 = too heavy at our scale.
- **Env isolation:** repo `.venv` pins (`transformers<5` for step91, gymnasium, torch) — the 3-D
  lane gets its **own venv** to avoid fighting swm pins.
